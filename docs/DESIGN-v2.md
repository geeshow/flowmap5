# flowmap 기능 확장 UI/UX 설계서

**대상 기능**: A. git commit 영향도 분석 · B. Kafka 영향도 분석 · C. RestDoc/API 문서 보기
**기준 코드**: `web/index.html` · `web/style.css` · `web/app.js` (v36) · `MANUAL.md`

---

## 0. 설계 원칙

1. **"뷰"가 아니라 "렌즈"** — 영향도(A·B)는 새 화면이 아니라 기존 컬럼 그래프 위에 얹는 하이라이트 오버레이로 설계한다. 사용자가 이미 아는 화면(호출관계분석, 서비스 보기)에서 그대로 켜고 끌 수 있어야 한다.
2. **기존 4개 표면 재사용** — 메인 캔버스(컬럼+SVG 커넥터), 오른쪽 상세 패널, 하단 프로세스 독, 글로벌 검색. 완전 신규 레이아웃 요소는 **왼쪽 레일(커밋 리스트) 하나**만 추가한다.
3. **URL = 화면 상태** 원칙 유지 — 모든 신규 상태는 URL 파라미터로 동기화하고, 공유 링크 복사가 그대로 동작한다.
4. **데이터는 lazy** — `impact.json`, `openapi.json`은 해당 기능 첫 진입 시에만 fetch. 없으면 기능을 숨기지 않고 **빈 상태로 안내**한다.

## 1. 기능 A — git commit 영향도 분석

### 1.1 진입점

| 진입 | 동작 |
|---|---|
| 헤더 nav에 **`🧾 커밋 영향도`** 버튼 추가 (기존 `🗺️ 전체보기` 옆) | `view=commits` 진입 |
| CONTROLLER 노드 상세 패널의 **"영향 커밋 N건"** 섹션 | 엔드포인트 관점 역조회 (`view=commits&ep=<nodeId>`) |
| 글로벌 검색에 커밋 인덱싱 (shortSha / subject / author) | 결과 그룹 `── 커밋 ──` 아래 표시 |

### 1.2 와이어프레임 — 커밋 영향도 뷰 (`view=commits`)

```
┌────────────────────────────────────────────────────────────────────────────┐
│ flowmap  [🗺️ 전체보기] [🧾 커밋 영향도●]   [ API 검색… ]      [🔗 공유 링크] │
├──────────────┬──────────────────────────────────────────────┬──────────────┤
│ 커밋 레일     │  🧾 커밋 영향도 › a1b2c3d (+1)                │ 상세 패널     │
│ ┌──────────┐ │  ┌─ 분석 바 ─────────────────────────────────┐│ (노드 선택 시)│
│ │필터: 작성자│ │  │ [a1b2c3d ✕][e4f5a6b ✕]  영향 합집합 ·     ││              │
│ │/메시지/파일│ │  │ 변경 4 · 영향 엔드포인트 7  [전체 해제]    ││ ◆ 영향 커밋  │
│ ├──────────┤ │  └────────────────────────────────────────────┘│  a1b2c3d     │
│ │☑ a1b2c3d │ │                                                │  e4f5a6b     │
│ │ 주문 검증 │ │   영향 엔드포인트      경로            변경 노드 │              │
│ │ 로직 수정 │ │  ┌───────────┐   ┌──────────┐   ┌───────────┐│              │
│ │ kyutae    │ │  │POST /orders◇──│OrderSvc  │◀──│validate() ◆│              │
│ │ 6/12      │ │  └───────────┘   │.create() │   └───────────┘│              │
│ │ 변경2·영향4│ │  ┌───────────┐   └──────────┘   ┌───────────┐│              │
│ ├──────────┤ │  │GET /orders ◇──── … ◀──────────│OrderRepo ◆│              │
│ │☑ e4f5a6b │ │  └───────────┘                   │.findBy()  │              │
│ │☐ 9c8d7e6 │ │       (◇=영향 엔드포인트  ◆=변경 노드)          │              │
│ └──────────┘ │                                          [줌⊕]│              │
└──────────────┴──────────────────────────────────────────────┴──────────────┘
```

- **레이아웃 방향은 기존 호출관계분석과 동일**: 피호출(caller)이 왼쪽. 변경 노드(◆)가 오른쪽 기준 컬럼, BFS 역추적 경로가 가운데, 영향받는 API 엔드포인트(◇)가 왼쪽 끝 컬럼. "코드를 바꾸면 ← 이 API들이 흔들린다"가 시선 흐름과 일치.
- **커밋 레일** (왼쪽, 폭 264px, 드래그 리사이즈 — `#detail-resizer` 패턴 미러링):
  - 커밋 카드 = `[☑] a1b2c3d`(mono 배지) + subject(1줄 말줄임) + `author · M/D` + `변경 N · 영향 M` 칩.
  - 체크박스 = **다중 선택(묶어 보기)**. 그래프는 선택 커밋들의 changedNodes/impactedEndpoints **합집합**을 그린다.
  - 카드 본문 클릭 = 단일 선택(기존 선택 대체), 체크박스 클릭 = 추가/제거. 스타일은 `.search-item` + `.node-card` 토큰 재사용.
- **그래프 본문은 기존 호출관계분석 렌더러 재사용** — 기준 노드가 1개가 아니라 changedNodes 복수라는 점만 다름(기준 컬럼에 변경 노드들을 세로 나열, `.column.imp-base` 주황 박스 변형). 커넥터·정렬 애니메이션·줌은 그대로.

### 1.3 엔드포인트 관점 역조회

별도 화면을 만들지 않는다. **어느 뷰에서든** CONTROLLER 노드를 선택하면 상세 패널 기존 테이블 아래에 섹션 추가:

```
│ ───────────────────────────│
│ ◆ 영향 커밋 3건              │
│  a1b2c3d 주문 검증 로직 수정  │
│  e4f5a6b 재고 차감 분리       │
│  [🧾 커밋 영향도에서 보기 →]  │
```

- 커밋 행 **hover** → 현재 캔버스에 그 커밋의 변경/영향 노드를 즉시 주황 프리뷰 하이라이트. 클릭 → `view=commits&commit=<sha>&ep=<nodeId>`.
- 영향 커밋 없는 엔드포인트는 섹션 자체를 렌더하지 않는다(빈 섹션 금지).

### 1.4 오버레이 모드

`commit=<sha>` 파라미터는 **호출관계분석/서비스 보기 위에서도 동작**: 현재 그래프에서 변경/영향 노드만 주황 처리, 나머지는 기존 `.dim`. 분석 바에 `◆ a1b2c3d 오버레이 [✕]` 칩. "이 서비스 화면에서 이번 커밋이 건드린 곳"을 신규 화면 없이 해결.

## 2. 기능 B — Kafka 영향도 분석

### 2.1 진입점

| 진입 | 동작 |
|---|---|
| 인프라 타입 보기(`infra=kafka`) 토픽 카드의 **`📡 영향도`** 호버 액션 | `view=topic&topic=<name>` |
| 토픽 노드 상세 패널의 **`📡 토픽 영향도 분석`** 버튼 | 〃 |
| 글로벌 검색 토픽 결과 행의 `📡` 보조 버튼 | 〃 |
| 호출관계분석/프로세스 독의 kafka 토픽 카드 **`consumer N`** 배지 클릭 | 〃 |

### 2.2 와이어프레임 — 토픽 영향도 뷰 (`view=topic`)

```
┌────────────────────────────────────────────────────────────────────────────┐
│  📨 Kafka › orders.created                                  [🔗 공유 링크]  │
│  ┌─ 분석 바 ──────────────────────────────────────────────────────────────┐│
│  │ ⟵ Kafka 토픽 목록   📨 orders.created                                  ││
│  │ producer 2 서비스 · consumer 3 서비스   ☑ 엔드포인트까지 역추적(end-to-end)││
│  └────────────────────────────────────────────────────────────────────────┘│
│   ── PRODUCE ⟶ ──                ╔══════════╗   ── ⟶ CONSUME ──            │
│   ┌ 🧩 order-service ──────────┐  ║          ║   ┌ 🧩 stock-service ──────┐│
│   │ ◇POST /orders → OrderSvc   │┄▶║ 📨 orders ║┄▶│ StockListener → StockSvc││
│   │   .create() → EventPub.send│  ║ .created ║   │   .decrease() → 🗄 stock││
│   └────────────────────────────┘  ║          ║   └─────────────────────────┘│
│   ┌ 🧩 admin-service ──────────┐  ║          ║   ┌ 🧩 noti-service ───────┐│
│   │ ◇POST /admin/orders → …    │┄▶║          ║┄▶│ NotiListener → 🌐 FCM   ││
│   └────────────────────────────┘  ╚══════════╝   │   ↳ 📨 noti.sent ▶ …   ││
│                                                   └─────────────────────────┘│
│        ◇ = 이 이벤트를 발행시키는 API    ↳ = 2차 토픽 체인 (이어서 펼침)  [줌⊕] │
└────────────────────────────────────────────────────────────────────────────┘
```

- **렌더러는 호출관계분석 재사용**, 차이 세 가지:
  1. 기준 노드 = 토픽(중앙 카드 1.25배 확대, `--c-kafka` 2px 외곽).
  2. **깊이 스테퍼 대신 "엔드포인트까지 역추적" 토글** — 켜면 produce 측 BFS를 CONTROLLER까지, consume 측 BFS를 leaf 또는 다음 토픽까지 자동 진행. 기존 200호출 상한·`(일부만 표시)` 규칙 적용.
  3. 좌우 컬럼을 **서비스 단위 박스**로 묶음 — 프로세스 독의 `.dock-seg` 패턴을 캔버스로.
- **produce / consume 시각 구분**: 보라 단일 색에서 채움으로 구분 —
  - produce 측 헤더: 보라 **외곽선** 칩 `PRODUCE ⟶`, 배경 틴트 `rgba(192,38,211,.04)`
  - consume 측 헤더: 보라 **채움** 칩 `⟶ CONSUME`, 배경 틴트 `rgba(192,38,211,.09)`
  - 엣지는 기존 규칙(보라 점선 + `kafka:produce/consume · async` 라벨) 그대로 — 새 색 도입 없음.
- **2차 토픽 체인**: consumer가 또 다른 토픽을 produce하면 consume 컬럼 오른쪽에 미니 토픽 카드(`↳`)로 이어 그리고, 클릭 시 그 토픽으로 재중심. 순환(A→B→A)은 `↺ 순환` 배지로 표시하고 더 펼치지 않음.

### 2.3 토픽 목록/검색

신규 목록 화면 없음. **기존 인프라 타입 보기(`infra=kafka`)가 토픽 목록**:
- 토픽 박스 헤더에 `▶ producer N · consumer M` 카운트 (consumer 0이면 빨강 틴트 — 죽은 토픽 가시화).
- 분석 바에 토픽 필터 입력(`#svc-filter` 재사용).
- 토픽 카드 호버 액션: `중심 ⟲`(기존) 옆에 `📡 영향도`(end-to-end) 추가.

### 2.4 "이 API를 바꾸면 어떤 컨슈머가 영향받나" 플로우

```
검색/브라우저에서 API 선택 → 호출관계분석 (focus=…#createOrder)
  → down 체인에 📨 orders.created 카드 등장 (consumer 3 배지)
  → 배지 클릭 → view=topic&topic=orders.created
  → consume 측에서 영향 컨슈머 확인 → 노드 클릭 → 상세 패널 → 🎯 재분석
```

핵심: 호출관계분석의 kafka 토픽 카드에 **consumer 수 배지를 상시 노출** — 배지가 곧 토픽 뷰로 가는 문.

## 3. 기능 C — RestDoc / API 문서 보기

### 3.1 상세 패널 탭 구조 (핵심 재사용)

operationId == 노드 id이므로 **상세 패널이 곧 문서 패널**이다. CONTROLLER 노드 선택 시 탭 추가:

```
┌ 상세 패널 (≥380px 권장) ────────────────┐
│ [ 개요 ] [ API 문서 ]                    │  ← openapi 매칭 시에만 탭 표시
│ ─────────────────────────────────────── │
│ POST /v1/orders            ── 14px bold │
│ 주문을 생성한다. 재고 확인 후 …  ── desc  │
│ ▸ 파라미터                               │
│ ┌──────────┬─────┬────────┬───────────┐ │
│ │ name     │ in  │ type   │ 설명       │ │
│ │ storeId* │query│ int64  │ 매장 ID    │ │
│ └──────────┴─────┴────────┴───────────┘ │
│ ▸ Request Body  application/json         │
│   OrderCreateRequest                     │
│   ├ items*      array<OrderItem>  [+]    │ ← $ref 클릭 전개
│   │  ├ skuId*   string            SKU    │
│   │  └ qty*     int32 (≥1)        수량   │
│   └ memo        string            메모   │
│ ▸ Responses                              │
│   [200] OrderResponse           [+]      │
│   [409] ErrorResponse · 재고 부족  [+]    │
│ ─────────────────────────────────────── │
│ [🎯 이 노드 기준 호출관계분석]            │ ← 문서→그래프
│ [🔗 이 화면 공유 링크 복사]               │
└──────────────────────────────────────────┘
```

- **개요 탭** = 현재 `renderDetail()` 출력 그대로. **API 문서 탭** = OpenAPI 렌더. 마지막 선택 탭은 localStorage에 기억.
- **스키마 트리**: 들여쓰기 트리(`.flow-row` 톤 재사용). 행 = `필드명* · 타입 · 제약 · 설명`. `$ref`는 `[+]` 클릭 시 **인라인 전개**. 순환 참조는 `↺ 재귀` 배지. 깊이 3 초과는 접힌 채 시작.
- **required**: 필드명 뒤 `*`(빨강 `--c-external`). enum은 값 칩 나열(최대 6개 + `+N`).
- 문서 탭 진입 시 패널 폭 380px 미만이면 한 번 자동 확장(사용자가 줄이면 존중, 별도 키에 저장).

### 3.2 API 카탈로그 뷰 (`view=api`)

진입 브라우저의 그리드/필터 컴포넌트 재사용, 카드 클릭 목적지만 다름(그래프 ✕ → 문서 ○).

```
┌────────────────────────────────────────────────────────────────────────────┐
│  📖 API 문서 › order-service                                                │
│  [ 경로/메서드/설명 필터…___________ ]                 42 endpoints          │
│  /v1/orders ──────────────────────────────                   ┌────────────┐│
│   POST /v1/orders        주문 생성          ◆2               │ 상세 패널   ││
│   GET  /v1/orders/{id}   주문 단건 조회                       │ (문서 탭    ││
│  /admin ──────────────────────────────────                   │  고정)     ││
│   POST /admin/refunds    환불 처리 (deprecated̶)              │            ││
└──────────────────────────────────────────────────────────────┴────────────┘
```

- 진입: 헤더 nav `📖 API 문서` 버튼, 또는 진입 브라우저 서비스 카드 호버 액션 `📖`.
- 행 클릭 → **목록 유지한 채** 오른쪽 상세 패널 문서 탭 열림(`sel=` + `tab=doc`). 마스터-디테일, 캔버스 전환 없음.
- `◆N` = 영향 커밋 수(A 기능과 자연 연동, impact 데이터 있을 때만).

### 3.3 그래프 ↔ 문서 양방향 이동

| 방향 | 동선 |
|---|---|
| 그래프 → 문서 | CONTROLLER 카드 호버 액션 `📄` (기존 `중심 ⟲` 옆) → 상세 패널 문서 탭 |
| 문서 → 그래프 | 문서 탭 하단 `🎯 호출관계분석` 버튼 |
| 검색 → 문서 | 검색 결과 행 `↗`(분석) 옆 `📄`(문서) 보조 버튼 |
| 카탈로그 → 그래프 | 행 호버 시 `⟲` 액션 |

### 3.4 description(한글 설명) 노출 위치

| 위치 | 표시 |
|---|---|
| 노드 카드 `.nc-desc` | 기존 유지 (1~2줄) |
| 상세 패널 개요 탭 | 기존 유지 |
| 문서 탭 | 본문 최상단, 전체 길이 |
| 카탈로그 행 | summary 1줄 말줄임 |
| 검색 | 기존처럼 매칭 대상 |

## 4. 진입 동선 & URL 파라미터 체계

### 4.1 헤더 nav 최종안

```
flowmap  [🗺️ 전체보기] [🧾 커밋 영향도] [📖 API 문서]   [ 검색… ]   [🔗 공유]
```

Kafka 영향도는 nav 버튼 없음 — 토픽은 "찾아 들어가는" 대상이며, 전체보기→Kafka 카드→토픽 동선이 이미 자연스럽다.

### 4.2 URL 파라미터 (기존 + 신규)

| 파라미터 | 의미 | 비고 |
|---|---|---|
| **`view=commits`** | 커밋 영향도 뷰 | |
| **`commit=<sha>[,<sha>…]`** | 선택 커밋 (CSV 다중) | `view=commits`에선 그래프 기준, **다른 뷰에선 오버레이** |
| **`ep=<nodeId>`** | 커밋 뷰의 엔드포인트 역조회 필터 | `view=commits` 전용 |
| **`view=topic` + `topic=<name>`** | 토픽 영향도 뷰 | `e2e=1`(기본) = 엔드포인트까지 역추적 |
| **`view=api`** | API 카탈로그 | 기존 `service=`/`path=` 재사용 |
| **`tab=doc`** | 상세 패널 문서 탭 열림 | `sel=`과 조합, 모든 뷰 공통 |
| 기존 13개 파라미터 | 변경 없음 | 신규 키와 중복 없음 (검증 완료) |

`parseUrl()` 우선순위: `view=commits` > `view=topic` > `view=api` > 기존 체인. 모르는 view 값은 기존처럼 무시 → 하위호환.

### 4.3 뒤로가기/Esc 체인 확장

- 신규 뷰 3종에서 `Esc` = 진입 출처로 복귀(기존 `from=`/`fo=` 패턴 재사용).
- 브레드크럼: `🧾 커밋 영향도 › a1b2c3d (+1)` / `🗺️ 전체보기 › 📨 Kafka › orders.created` / `📖 API 문서 › order-service › POST /v1/orders`.

## 5. 시각 언어 확장

### 5.1 영향도 하이라이트 (신규 디자인 토큰)

기존 의미색: 앰버 `#fbbf24` = 기준, 스카이 `#38bdf8` = 선택, 빨강 = external, 보라 = kafka, 핑크 = redis, 갈색 = db. → **주황이 유일하게 비어 있는 의미 슬롯**이므로 영향도 전용 배정.

```css
:root {
  --imp-changed:  #f97316;                 /* 변경 노드 */
  --imp-tint:     rgba(249,115,22,.10);    /* 영향 경로 노드 배경 틴트 */
  --imp-halo:     rgba(249,115,22,.35);    /* 영향 경로 엣지 halo */
}
```

| 대상 | 스타일 |
|---|---|
| **변경 노드 ◆** | `box-shadow: 0 0 0 2px var(--imp-changed)` 실선 링 + `◆ COMMIT a1b2c3d` 주황 채움 배지 (mono) |
| **영향 엔드포인트 ◇** | 2px **점선** 주황 링 + `◇ 영향` 외곽선 배지 |
| **영향 경로 중간 노드** | 링 없음 — 배경 `--imp-tint`만. 레이어 좌측 보더 유지 |
| **영향 경로 엣지** | 기존 kind 색을 **덮지 않는다**. 같은 path 2회 렌더: 아래 stroke 7px `--imp-halo`, 위에 기존 kind 색 → "형광펜 밑줄" 효과로 sync/async·kind 정보 보존 |
| **비영향 노드/엣지** | 기존 `.dim` 재사용 |

**충돌 회피**: 기준(앰버)·선택(스카이)·변경(주황) 링이 한 노드에 겹칠 수 있음 → 링은 한 번에 하나(우선순위: 선택 > 기준 > 변경), 밀려난 의미는 배지로 강등. 색이 아니라 채널(링 vs 배지)을 분리해 해결.

### 5.2 커밋 배지/칩

- shortSha는 항상 `ui-monospace` 10px, 주황 1px 보더 칩 `◆ a1b2c3d`. 모든 위치에서 동일 형태.
- 커밋 레일 선택 카드: 기존 `.search-item.kb-active`의 좌측 보더 패턴을 주황으로.

### 5.3 Kafka produce/consume 구분

신규 색 없음 — `--c-kafka` 단일 색에서 **채움 vs 외곽선**으로 구분. 색약 사용자를 위해 칩 텍스트에 방향 화살표(`PRODUCE ⟶` / `⟶ CONSUME`) 항상 포함.

### 5.4 API 문서 패널 타이포그래피

| 요소 | 스타일 |
|---|---|
| 메서드+경로 헤더 | 14px 700, 경로 `ui-monospace`, 메서드는 기존 `.nc-badge.m-*` 색 |
| summary | 13px 600 `--text` |
| description 본문 | 12px, line-height 1.7, `#cbd5e1` (기존 `.dn-desc` 색) |
| 섹션 헤더 (`▸ 파라미터`) | 10px 700 uppercase — 기존 `.column-head` 토큰 |
| 파라미터 테이블 | 기존 `.detail-table` 11px 재사용 |
| 스키마 트리 행 | 11px, 필드명 `--text` / 타입 `#7dd3fc` mono / required `*` 빨강 |
| 상태코드 배지 | 2xx 초록 · 4xx 앰버 · 5xx 빨강 |

## 6. 인터랙션 디테일

### 6.1 Hover / 클릭 / 키보드

| 컨텍스트 | 인터랙션 |
|---|---|
| 커밋 레일 카드 hover | 120ms 디바운스 후 캔버스에 해당 커밋 **프리뷰 하이라이트**(벗어나면 원복) |
| 커밋 레일 키보드 | `↑↓` 이동, `Space` 체크 토글, `Enter` 단일 적용. `/`는 항상 글로벌 검색 점유(기존 규칙 유지) |
| 영향 엔드포인트 ◇ 클릭 | 상세 패널 + "영향 커밋" 섹션 자동 펼침 |
| 토픽 카드 consumer 배지 | hover 시 컨슈머 서비스명 툴팁, 클릭 시 토픽 뷰 |
| 스키마 트리 `[+]` | 클릭 인라인 전개/접기, `←/→`로도 가능 |
| 문서 탭 ↔ 개요 탭 | 클릭 또는 `[`/`]`. `Esc`는 탭을 건드리지 않고 기존 체인 유지 |
| 줌/리사이즈 | 전 신규 뷰에서 기존 줌·핀치·리사이저 동일 동작 |

### 6.2 빈 상태 (기존 `.flow-empty` / `.browse-empty` 패턴 재사용)

| 상황 | 표현 |
|---|---|
| `impact.json` 404 | `🧾 커밋 영향도 데이터가 없습니다` + 생성 명령어 안내 코드 블록 + `[전체보기로]`. **nav 버튼은 숨기지 않는다** |
| 커밋은 있으나 영향 엔드포인트 0 | `변경 노드 N개가 어떤 엔드포인트에도 닿지 않습니다` + 변경 노드 목록 |
| 토픽에 consumer 없음 | 점선 보더 빈 카드 `컨슈머 없음 — 이 이벤트는 소비되지 않습니다` |
| `openapi.json` 404 / operationId 미매칭 | 문서 탭 자체를 렌더하지 않음(개요 탭만) |
| 스키마 `$ref` 해석 실패 | `⚠ 스키마 누락: <ref>` dim 표기, 트리 나머지는 정상 렌더 |

### 6.3 로딩

- `impact.json`/`openapi.json`은 첫 진입 시 fetch. **200ms 안에 오면 로딩 UI 생략**, 넘으면 `불러오는 중…` + 카드 스켈레톤 3장(opacity pulse만).
- fetch 실패는 빈 상태 컴포넌트에 `다시 시도` 버튼 추가.
- 토픽 뷰 end-to-end가 200호출 상한 도달 시 기존 `(일부만 표시)` 라벨 재사용.

## 7. 기존 기능 충돌 점검표

| 점검 항목 | 판정 | 근거/조치 |
|---|---|---|
| URL 파라미터 키 충돌 | **없음** | 신규 키 `commit/ep/topic/e2e/tab`은 기존 13개와 미중복 |
| 기존 공유 링크 하위호환 | **유지** | 기존 파라미터 의미 불변 |
| `Esc`/`Backspace` 체인 | **확장만** | 신규 뷰 3종을 체인 맨 앞에 추가, 기존 순서 불변 |
| `/` 검색 단축키 | **유지** | 커밋 레일 필터는 별도 input |
| 색 의미 충돌 | **해소** | 주황은 미사용 슬롯. 링 1개 + 배지 강등 규칙 |
| 엣지 kind 색 보존 | **보존** | halo 이중 렌더 — 덧칠 아님 |
| 상세 패널 | **확장만** | 탭은 CONTROLLER+OpenAPI 매칭 시에만 |
| 프로세스 독 | **충돌 없음** | 독은 "서비스 보기 3단계 전용" 유지. 커밋 리스트를 하단 독이 아닌 좌측 레일로 둔 이유 |
| localStorage | **분리** | 신규 키 3개, 기존 키 미접촉 |
| 줌/커넥터 렌더러 | **재사용** | 신규 뷰 전부 `#zoom-layer` + `drawConnectors` 경로 |
| 검색 | **확장만** | 커밋·토픽 결과는 그룹 헤더로 구분, 40건 상한·키보드 탐색 불변 |
| 기존 인프라 타입 보기 | **확장만** | 클릭 기본 동작 불변 — 영향도는 명시적 `📡` 액션으로만 |
| 성능 | **주의 1건** | 커밋 다중 선택 합집합 + end-to-end는 노드 폭증 가능 → 기존 200호출 상한 공통 가드 |

## 8. 구현 우선순위 제안

1. **C-1 상세 패널 문서 탭** — 신규 화면 0개, 체감 가치 최대.
2. **A 커밋 뷰 + 오버레이** — 좌측 레일 1개 + 기존 렌더러 멀티-기준 확장.
3. **B 토픽 영향도 뷰** — A에서 만든 멀티-기준/end-to-end BFS 인프라 재사용.
4. **C-2 카탈로그 뷰** — 진입 브라우저 그리드 재사용, 마지막에 저비용 부착.
